01 — Overview

What Chuckle is

Chuckle is a question-answering system for baby activity data. A parent uploads a CSV exported from the Huckleberry app, then types questions in plain English. The system translates those questions into SQL, runs them against a local SQLite database, and returns a natural-language answer.

No data leaves the machine. There is no server — the LLM is called via OpenRouter, but the database stays local.

UI layer

Streamlit

Browser-based chat interface with a sidebar file uploader. Runs locally via uv run streamlit run app.py.

Agent framework

LangChain

create_tool_calling_agent + AgentExecutor — the standard structured tool-use loop.

Language model

OpenRouter → GPT-4o-mini

Any OpenAI-compatible model via OpenRouter. Default: openai/gpt-4o-mini. Overridable with CHUCKLE_MODEL.

Storage

SQLite + events table

Single-file database (chuckle.db) with one denormalised table and two indexes. Replaced wholesale on each CSV upload.

02 — Mechanism

The tool-calling loop

Chuckle uses a ReAct-style agent — the LLM alternates between reasoning and acting until it has enough information to answer. LangChain's create_tool_calling_agent implements this via the model's native function-calling API rather than text parsing.

The tool-calling loop — solid lines are the outbound request, dashed lines are the return path

1

LLM receives the question

The system prompt (with live DDL) and the user's question are assembled into a ChatPromptTemplate. The LLM sees the full database schema and a {agent_scratchpad} placeholder where previous tool results accumulate.

2

LLM decides to call a tool

When the model needs data, it emits a tool call — a structured JSON object with name: "query_database" and an sql argument. This is not free-text; it uses the model's native function-calling API (the same mechanism as OpenAI function calling).

3

AgentExecutor dispatches the tool

LangChain's AgentExecutor catches the tool call, validates the name against the registered tool list, and invokes query_database(sql=...). The function checks the SQL starts with SELECT, runs it via db.run_select(), and returns a JSON array of rows.

4

Observation is appended to the scratchpad

The rows are added as a tool message in the conversation. The model now sees: original question → its own tool call → the result. It can call another tool, or if it has enough data, produce the final answer.

5

LLM generates the final answer

When the model emits a plain message (no tool call), AgentExecutor treats it as the final answer and returns it. The loop is bounded by max_iterations=3 to prevent runaway queries.

Why tool calling, not text parsing? Earlier ReAct implementations parsed "Thought / Action / Observation" from raw LLM output — fragile and prompt-sensitive. Modern models expose a dedicated function-calling API: the tool call is a typed JSON object, not text. LangChain's create_tool_calling_agent uses this API, making the loop reliable even without chain-of-thought prompting.

03 — LangChain implementation

The code, file by file

Four Python modules. Each has one job.

db.py — the data layer

Creates and owns the SQLite connection. Defines the schema, enforces column constraints, and provides the single read helper used by the agent tool.

db.py — run_select (the only entry point the agent uses) 
def run_select(conn: sqlite3.Connection, sql: str) -> list[dict]:
    stripped = sql.strip().lower()
    if not stripped.startswith("select"):
        raise ValueError("only SELECT statements are allowed")
    cursor = conn.execute(sql)
    return [dict(row) for row in cursor.fetchall()]

The guard is a simple string prefix check — not a full SQL parser. It blocks INSERT/UPDATE/DELETE at the application layer; the database itself is also opened without WAL or write permissions in production.

The schema is defined as a CREATE TABLE IF NOT EXISTS statement run at startup via init_db(). The table is fully denormalised — all event types share one table, with NULL columns for fields that don't apply.

ingest.py — CSV → SQLite

Parses the Huckleberry export. Currently handles Feed rows only (TB-1 scope). Each row is validated, normalized, and returned as a dict matching the events table columns.

ingest.py — feed side parsing 
# CSV encodes breast time as e.g. "00:05R" (5 min right)
_FEED_SIDE_RE = re.compile(r"^(\d{1,2}):(\d{2})([LR])$", re.IGNORECASE)

def _parse_feed_side(value: str | None) -> tuple[str | None, int]:
    if not value:
        return None, 0
    match = _FEED_SIDE_RE.match(value.strip())
    if not match:
        return None, 0
    hours, minutes, side = match.groups()
    total_minutes = int(hours) * 60 + int(minutes)
    return side.upper(), total_minutes

Start Condition and End Condition columns both contribute to total side time. They're summed: if a feed starts right for 5 min and ends right for 3 min, feed_right_minutes = 8.

On upload, the entire events table is replaced via db.replace_events() — DELETE then bulk INSERT in a single transaction. There is no incremental merge.

agent.py — the LangChain agent

The core wiring. Three LangChain objects are constructed and composed.

agent.py — build_agent 
def build_agent(now: datetime, conn: sqlite3.Connection | None = None) -> AgentExecutor:
    conn = conn or db.init_db()

    # 1. The model — temperature=0 for deterministic SQL generation
    llm = ChatOpenAI(
        model=model_name,
        api_key=api_key,
        base_url=OPENROUTER_BASE_URL,
        temperature=0,
    )

    # 2. The tool — a Python function decorated with @tool
    tools = _build_tools(conn)

    # 3. The prompt — system + human turn + scratchpad placeholder
    prompt = ChatPromptTemplate.from_messages([
        ("system", build_system_prompt(_live_ddl(conn))),
        ("human", "{input}"),
        ("placeholder", "{agent_scratchpad}"),
    ])

    # 4. Compose into an agent + executor
    agent = create_tool_calling_agent(llm, tools, prompt)
    return AgentExecutor(agent=agent, tools=tools, max_iterations=3)
Object LangChain class Role
llm ChatOpenAI Wraps the model API. temperature=0 makes SQL generation deterministic.
tools @tool decorator Converts query_database(sql) into a LangChain StructuredTool with auto-generated JSON schema from the type signature.
prompt ChatPromptTemplate Three slots: system (static), human input (question), and scratchpad (accumulates tool call/result pairs each iteration).
agent create_tool_calling_agent Binds the LLM + tools + prompt. Handles the function-calling API format automatically.
executor AgentExecutor Runs the loop: invoke agent → dispatch tools → append observation → repeat until done or max_iterations.
agent.py — the @tool definition 
def _build_tools(conn: sqlite3.Connection) -> list:
    @tool
    def query_database(sql: str) -> str:
        """Run a single read-only SQL SELECT against the events table and
        return the rows as a JSON array. Use this for any question about
        the baby's activity data. Only SELECT is allowed."""
        try:
            rows = db.run_select(conn, sql)
        except (ValueError, sqlite3.Error) as exc:
            return f"ERROR: {exc}"
        return json.dumps(rows, default=str)

    return [query_database]
Why is the tool defined inside a function? The conn (SQLite connection) is captured via closure. This avoids passing the connection as a global and lets the tool be injected with a test connection during testing — _build_tools(test_conn).

prompts.py — system prompt

Assembles the system prompt at runtime by injecting the live DDL. The LLM sees the actual table structure — column names, types, and CHECK constraints — every time.

prompts.py — build_system_prompt 
ROLE_BLOCK = (
    "You are Chuckle, a helpful assistant that answers questions about a baby's "
    "activity data. You have access to a SQLite database via the `query_database` "
    "tool, which runs read-only SQL SELECT statements against an `events` table. "
    "Use it whenever the question requires data; respond in plain natural language "
    "and do not mention SQL or the database in your answer."
)

def build_system_prompt(schema_ddl: str) -> str:
    return f"""{ROLE_BLOCK}

Database schema (live DDL):
{schema_ddl}
"""

The DDL is fetched fresh from sqlite_master each time build_agent() is called — so the prompt always reflects the actual schema, not a hardcoded copy. If the schema changes, the prompt updates automatically.

04 — End-to-end walkthrough

A concrete example

Question: "How many times did Leo feed today?"

#What happensWhere in the code
1 User types in the Streamlit chat box. app.py appends the message and calls agent.answer(question, now=datetime.now()). app.py:79
2 answer() calls build_agent(), which creates the LLM, tool, and prompt, then returns an AgentExecutor. agent.py:51–78
3 executor.invoke({"input": question}) sends the system prompt + question to the model. The system prompt includes the live DDL. agent.py:83
4 The model emits a tool call: {"name": "query_database", "arguments": {"sql": "SELECT COUNT(*) FROM events WHERE type='Feed' AND DATE(start_time)=DATE('now')"}}. Model API response
5 AgentExecutor dispatches query_database(). The prefix guard passes. db.run_select() executes the SQL and returns [{"COUNT(*)": 7}]. agent.py:38–46, db.py:114
6 The JSON result is appended to the scratchpad as a tool message. The model is called again with the full context. AgentExecutor internals
7 Model emits a plain text reply: "Leo fed 7 times today." No tool call — the loop ends. Model API response
8 result["output"] is returned to app.py, displayed in the chat, and stored in st.session_state.messages. agent.py:84, app.py:80
Note on "today": The SQL uses SQLite's DATE('now'), which is UTC. TB-1 does not pass the user's local time to the agent — temporal context (converting "today", "this morning", "last Wednesday" to absolute timestamps) is planned for TB-3.

05 — Current limits & what's next

TB-1 scope

This is a walking skeleton — the minimal end-to-end system. Several things are deliberately deferred.

TB-1 · now

Feed rows only

ingest.py silently skips all non-Feed rows. Sleep, Diaper, Bath etc. are not yet ingested.

TB-2 · next

All event types

Implement the remaining 8 parsers (Sleep, Diaper, Bath, Tummy time, Pump, Meds, Growth, Temp).

TB-1 · now

No temporal context

The agent doesn't know the user's local time. "Today" and "this morning" resolve to SQLite UTC, which may be wrong.

TB-3 · next

Temporal resolution

Pass now into the system prompt. Map "yesterday", "Wednesday", "last week" to absolute datetime ranges.

TB-1 · now

Minimal system prompt

Role + raw DDL only. No column descriptions, no sample rows, no few-shot examples.

TB-3 · next

Richer prompt

Add column semantics, sample values, few-shot query/answer pairs, and temporal reasoning guidance.

TB-1 · now

No streaming

Responses appear all at once when the full answer is returned. The UI blocks during LLM calls.

TB-3 · next

Streaming output

Use executor.astream_events() + Streamlit's st.write_stream() for token-by-token display.
